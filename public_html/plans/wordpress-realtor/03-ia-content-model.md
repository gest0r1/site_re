## 3. Lead-формы: поля, consent, файлы, состояния, Bitrix24

### 3.1 Общая спецификация всех форм

| Параметр | Значение |
|----------|----------|
| **Плагин форм** | Fluent Forms (или CF7 + Bitrix24-мост) |
| **Anti-spam** | Cloudflare Turnstile (invisible) |
| **Honeypot** | да (скрытое поле, заполняется только ботами) |
| **Rate limiting** | 3 попытки с одного IP за 60 секунд (через плагин или nginx) |
| **Consent checkbox** | обязателен на каждой форме, текст: «Согласен на обработку персональных данных согласно [Политике](/privacy-policy/)» |
| **Валидация на клиенте** | HTML5 + JS (formidable) |
| **Валидация на сервере** | PHP (все поля) |
| **Без JS** | HTML5 validation работает, капча fallback на Turnstile non-interactive |
| **Success action** | Показать success-страницу (inline) + отправить лид в Bitrix24 |
| **Failure action** | Показать ошибку + retry 3 раза + уведомление админу email |
| **Дубликаты** | По телефону: если телефон уже есть в Bitrix24 — не создавать дубль, показать «Мы уже получили вашу заявку» |

### 3.2 Формы: поля и привязка к Bitrix24

#### Форма A: Seller Diagnostic (`/sell/diagnostic/`)

| Поле | Тип | Обязательное | Валидация | Bitrix24 поле (title) |
|------|-----|-------------|-----------|----------------------|
| `name` | text | да | min 2 символа, только буквы/дефис | `NAME` |
| `phone` | tel | да | +7 (XXX) XXX-XX-XX, 11 цифр | `PHONE` |
| `address` | text | да | min 10 символов | `ADDRESS` |
| `area_total` | number | да | > 0 | `UF_CRM_AREA` |
| `floor` | number | да | ≥ 1 | `UF_CRM_FLOOR` |
| `floors_total` | number | да | ≥ floor | `UF_CRM_FLOORS_TOTAL` |
| `building_type` | select | да | panel/monolith/brick/block/wood | `UF_CRM_BUILDING_TYPE` |
| `condition` | select | да | requires-repair/cosmetic/good/excellent | `UF_CRM_CONDITION` |
| `photos` | file upload | нет | JPEG/PNG, max 10MB каждый, макс 5 файлов | не передаётся |
| `consent` | checkbox | да | must be checked | не передаётся |
| `source_page` | hidden | да | автозаполнение | `UF_CRM_SOURCE_PAGE` |
| `form_type` | hidden | да | diagnostic | `UF_CRM_FORM_TYPE` |

**Bitrix24 payload:**
```json
{
  "fields": {
    "TITLE": "Лид: Seller Diagnostic — {address}",
    "NAME": "{name}",
    "PHONE": [{ "VALUE": "{phone}", "VALUE_TYPE": "WORK" }],
    "UF_CRM_ADDRESS": "{address}",
    "UF_CRM_AREA": "{area_total}",
    "UF_CRM_FLOOR": "{floor}",
    "UF_CRM_FLOORS_TOTAL": "{floors_total}",
    "UF_CRM_BUILDING_TYPE": "{building_type}",
    "UF_CRM_CONDITION": "{condition}",
    "UF_CRM_SOURCE_PAGE": "{source_page}",
    "UF_CRM_FORM_TYPE": "diagnostic",
    "SOURCE_ID": "WEB",
    "SOURCE_DESCRIPTION": "Seller Diagnostic — {source_page}"
  }
```

## 4. AI workflow: статусы, роли, шаги, блокеры

### 4.1 AI governance: принципы

| Принцип | Правило |
|---------|---------|
| **Human-in-the-loop** | AI никогда не публикует контент автоматически. Только черновик. |
| **No auto-publish** | Любой AI-сгенерированный контент требует ручного action редактора. |
| **No PII in AI** | Персональные данные (имя, телефон, email) не передаются в AI-промпты. |
| **AI не гарантирует** | Disclaimer на каждом AI-результате: «Результат AI-анализа и не является юридической или оценочной гарантией.» |
| **Версионирование** | Каждый AI-черновик хранит `_ai_draft_version` post meta и `_ai_reviewed_by` при публикации. |

### 4.2 AI-статусы контента (post meta `_ai_status`)

| Статус | Описание | Кто устанавливает |
|--------|----------|------------------|
| `human` | Контент написан человеком, AI не участвовал | Редактор (по умолчанию) |
| `ai_draft` | AI-черновик, ожидает ревью | AI (автоматически) |
| `ai_reviewed` | Черновик проверен редактором, ожидает публикации | Редактор (вручную) |
| `ai_published` | Опубликовано после AI-цикла | Редактор |
| `ai_rejected` | Черновик отклонён, не будет опубликован | Редактор |
| `ai_blocked` | Черновик заблокирован (факт. ошибка, юр. риск) | Редактор |

### 4.3 Роли в AI-цикле

| Роль | Действия | Система |
|------|----------|---------|
| **AI Generator** | Создаёт черновик по теме + ключевым точкам | WordPress AI (Jetpack AI / AIOSEO / Writer plugin) |
| **Редактор** | Правит, даёт уточнения, проверяет факты, утверждает | Человек в wp-admin |
| **Чек-лист gate** | Блокирует публикацию до прохождения проверок | mu-plugin `ai-publication-gate.php` |

**Права по умолчанию:**

| Роль | Может | Не может |
|------|------|----------|
| **AI Generator** | Создавать `ai_draft`, предлагать правки | Публиковать, менять статусы вручную, видеть ПДн в промптах |
| **Редактор** | Редактировать, ревьюить, ставить `ai_reviewed`, публиковать вручную | Обходить gate, публиковать без чек-листа |
| **Чек-лист gate** | Валидировать факты/SEO/PII/дисклеймеры | Менять контент |

### 4.4 AI Editorial Workflow (шаги)

```
[1] Редактор → задаёт тему + 3–5 ключевых точек в форме AI Editor
    ↓
[2] AI → генерирует черновик (статус: ai_draft)
    ↓
[3] Редактор → читает, правит, даёт уточнение «Уточнить AI»
    ↓ (итерации по циклу 2→3)
[4] Редактор → запускает чек-лист перед публикацией
    ↓
[5] AI Gate mu-plugin: проверяет:
    - статус установлен в ai_reviewed
    - нет незакрытых блокеров (см. 4.5)
    - факт-чекинг пройден
    - SEO-теги заполнены
    - дисклеймер присутствует
    ↓ если всё ОК
[6] Редактор → публикует вручную (статус: ai_published)
    ↓ если блокер
[6b] Редактор → исправляет → повторный чек-лист → публикация
```

### 4.5 AI Publication Gate (mu-plugin `ai-publication-gate.php`)

Перед публикацией `_ai_status` = `ai_reviewed` → блокирующие проверки:

| Check | Что проверяет | Если не пройдено |
|-------|---------------|-----------------|
| `fact_check` | Факты проверены (чекбокс редактора) | Кнопка «Опубликовать» неактивна |
| `legal_disclaimer` | Текст содержит дисклеймер (если нужно по рубрике) | Предупреждение, публикация возможна с подтверждением |
| `seo_meta` | Заполнен `_yoast_wpseo_title` и `_yoast_wpseo_metadesc` | Предупреждение |
| `no_pii` | В контенте нет номеров телефонов/email (regex) | Блокер (автоматически) |
| `plagiarism` | Пройдена проверка уникальности (чекбокс) | Предупреждение |
| `client_approval` | Клиент утвердил (чекбокс для публичных статей) | Предупреждение |

## 7. Edge cases (граничные случаи)

| # | Сценарий | Реакция |
|---|----------|---------|
| EC-01 | Продавец загрузил не фото квартиры, а кота/еду | AI: «На фото не интерьер». После 3 неудач: CTA на консультацию |
| EC-02 | Продавец повторная заявка на тот же адрес | Проверка дубликатов по телефону → сообщение «Вы уже оставляли заявку» |
| EC-03 | Адрес в risk-check не найден в базе | «Мы не нашли адрес. Оставьте контакт — проверим вручную» → лид |
| EC-04 | Объект `sold`, покупатель заходит на карточку | CTA скрыта. Баннер «ПРОДАН». Блок похожих (2–3 available). Если нет — CTA консультация |
| EC-05 | Объект `archived` | Баннер «АРХИВИРОВАН». Блок похожих. CTA — консультация. |
| EC-06 | AI-диагностика выдала некорректные цифры | Human-in-the-loop: AI только черновик. Редактор проверяет перед отправкой |
| EC-07 | Bitrix24 недоступен при отправке лида | Retry 3× (интервал 5s). Все неудачны → лог + email админу |
| EC-08 | JavaScript отключён | HTML5-валидация форм работает. AJAX fallback на перезагрузку. Навигация работает. |
| EC-09 | Рейтинг застройщика < 30 | Отображается объективно. Дисклеймер. Объекты не блокируются. |
| EC-10 | Редактор публикует AI-статью без проверки фактов | mu-plugin `ai-publication-gate.php` блокирует. Чекбокс «факты проверены» обязателен. |
| EC-11 | Пустой результат поиска в каталоге | «Ничего не найдено. Измените фильтры или закажите подбор» → CTA |
| EC-12 | Покупатель из другого региона | Фильтры по умолчанию — Москва. Сообщение «Сейчас работаем по Москве» + CTA |
| EC-13 | Продавец хочет отозвать заявку | Форма «Отозвать заявку» → email админу → обработка в Bitrix24 вручную |
| EC-14 | Файл >10MB при загрузке фото | Сообщение: «Файл слишком большой. Максимум 10 MB.» |
| EC-15 | Turnstile не загрузился (блокировщик рекламы) | Fallback: текстовый вызов (checkbox «Я не робот») или одноразовый токен по email |
| EC-16 | Нет похожих объектов для sold (каталог пуст) | Вместо блока похожих — CTA «Закажите подбор — мы найдём похожий вариант» |

## 9. Навигация и CTA: правила

### 9.1 Структура меню

```
[Главная]

[Продаю]              [Покупаю]              [Инструменты]          [О нас]
├── Оценка онлайн     ├── Проверка рисков    ├── Глоссарий          ├── О компании
├── Проверка фото     ├── Каталог новостроек ├── FAQ                ├── Отзывы и кейсы
├── AI Diagnostic     ├── Каталог вторички   ├── Блог               └── Контакты

├── Торг и показы     └── Анализ районов

```

**Реализация:** два отдельных меню (locations) в WordPress: `menu-seller` и `menu-buyer`. На страницах seller-трека показывается `menu-seller`, на buyer-треке — `menu-buyer`. На главной — оба. В хедере — ссылки на оба трека. В футере — всё.

### 9.2 Правила CTA

| Условие | CTA |
|---------|-----|
| Seller на любой seller-странице | Основная: «Бесплатная оценка» → `/sell/estimate/`<br/>Secondary: «Полный диагностик» → `/sell/diagnostic/` |
| Buyer на любой buyer-странице | Основная: «Проверить риски» → `/buyers/risk-check/`<br/>Secondary: «Смотреть каталог» → `/catalog/new/` |
| Карточка объекта, status=available | «Записаться на просмотр» → форма E |

## 11. Управление статусами sold/archived

| Статус | Видимость в каталоге | Видимость в поиске | Карточка | CTA |
|--------|---------------------|-------------------|----------|-----|
| `available` | Да (фильтр по умолчанию) | Да | Полная | «Записаться на просмотр» |
| `under-contract` | Да (параметр show=all) | Да | Баннер «В резерве» + форма ожидания | «Сообщить при освобождении» |
| `sold` | Только при show=sold | Да (с пометкой) | Баннер «ПРОДАН» + похожие | Скрыта |
| `archived` | Только при show=archived | Нет (meta_query exclude) | Баннер «АРХИВИРОВАН» + похожие | Скрыта |

### 11.1 SEO-правила для sold/archived

| Статус | Robots | Canonical | Sitemap |
|--------|--------|-----------|---------|
| `sold` | `index,follow` | self canonical | включать, если страница полезна для поиска и кейсов |
| `archived` | `noindex,follow` | self canonical | не включать в основной sitemap |

**Примечание:** sold-страницы сохраняют SEO-ценность и кейсовую полезность; archived — не индексируем, но оставляем доступными по прямой ссылке и через внутренние подборки.

**Реализация:** `pre_get_posts` в mu-plugin `catalog-query.php`:
- По умолчанию: `meta_query` = `status` IN (`available`, `under-contract`)
- Для show=sold: добавить `sold`
- Для show=archived: добавить `archived`
- На карточке: показывать любой статус (проверка в `single-property.php`)

**Автоматизация:** не требуется. Редактор вручную меняет статус через ACF Select.
В будущем: wp-cron для автоматической архивации объектов старше 12 месяцев со статусом sold.
