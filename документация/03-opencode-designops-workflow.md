# Opencode DesignOps workflow

## Цель

Убрать ручную передачу промтов между ChatGPT Web и opencode.

ChatGPT Web используется для проектирования, уточнения требований и фиксации замечаний. Opencode используется для исполнения через доступ к WordPress, WP-CLI, SSH и файлам репозитория.

## Рабочая схема

```text
ChatGPT Web
  ↓
оформляет дизайн-требование или замечание
  ↓
/беклог-промтов/*.md
  ↓
opencode читает задачу
  ↓
проверяет текущий сайт
  ↓
вносит изменения
  ↓
делает скриншоты / отчёт
  ↓
пользователь подтверждает или возвращает замечания
```

## Папки

```text
/документация/      — стабильные правила, дизайн-контекст, ревью, карты ассетов
/беклог-промтов/    — атомарные задачи и замечания после проверки дизайна
```

## Что opencode должен читать перед задачей

Перед выполнением любого файла из `/беклог-промтов/` opencode должен прочитать:

1. `/документация/README.md`
2. `/документация/01-дизайн-контекст-и-правила-агента.md`
3. `/документация/02-инструкция-дизайн-ревью.md`, если задача связана с проверкой
4. `/документация/04-карта-ассетов.md`, если задача связана с изображениями, логотипами или иконками
5. текущий контекст структуры сайта, который уже находится в opencode

## Что opencode не должен делать без подтверждения

- менять структуру URL;
- менять header/footer menu вне утвержденной структуры;
- добавлять новые плагины;
- переходить с Kadence/Gutenberg на Elementor;
- заменять макет одной картинкой;
- публиковать неподтвержденные факты;
- создавать fake catalog при отсутствии реальных объектов;
- менять несколько крупных блоков в одной задаче.

## Формат задачи для opencode

Каждый файл в `/беклог-промтов/` должен содержать:

```md
# ID — Название задачи

## Статус

new / in-progress / done / rejected

## Источник замечания

Кто обнаружил: пользователь / ChatGPT Web / opencode reviewer / скриншот.

## Проблема

Что не соответствует дизайну.

## Где

URL, viewport, блок.

## Как должно быть

Конкретный ожидаемый результат.

## Что можно менять

Настройки Kadence, Gutenberg-блоки, CSS, шаблон, контент.

## Что нельзя менять

Ограничения.

## Проверка

Как понять, что задача выполнена.

## Отчет opencode

Заполняется после выполнения.
```

## Правило атомарности

Одна задача — один проверяемый результат.

Плохо:

```text
Довести главную до дизайна.
```

Хорошо:

```text
Исправить sticky header: при прокрутке фон становится белым, текст меню пропадает.
```

## Рекомендуемый цикл

1. Пользователь или ChatGPT Web фиксирует замечание в `/беклог-промтов/NNN-name.md`.
2. Opencode выполняет только этот файл.
3. Opencode не меняет сайт до чтения дизайн-документов.
4. Opencode делает минимальный план.
5. Opencode вносит изменения.
6. Opencode проверяет desktop/tablet/mobile.
7. Opencode пишет результат в конец того же backlog-файла.
8. Пользователь подтверждает или возвращает замечания.

## Команда для opencode

```text
Прочитай /документация/README.md и выполни /беклог-промтов/NNN-name.md. Структуру сайта бери из своего текущего контекста. Не переходи к следующей задаче без подтверждения пользователя.
```

## Отчет после выполнения

Opencode должен указать:

- что изменено;
- какие файлы/настройки затронуты;
- какие URL проверены;
- какие viewport проверены;
- какие проблемы остались;
- какие скриншоты приложены или где они сохранены;
- что требует подтверждения пользователя.

## Если проверка opencode ограничена

Если opencode не может надежно проверить визуальное соответствие, он должен:

1. Зафиксировать ограничение.
2. Сделать максимально доступную проверку.
3. Попросить пользователя приложить скриншот.
4. Не считать задачу полностью принятой без визуального подтверждения.

Именно для этого существует папка `/беклог-промтов/`: замечания после ручной проверки фиксируются там и затем исполняются opencode.
